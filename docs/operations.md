# Operations and deployment

## 1. Current deployment posture

The repository includes Foundry scripts for native and ERC-20 deployment, immutable-configuration
and runtime-hash auditing, dry-run and broadcast Make targets, a target-network catalog, an
unsigned-observation generator, and separate observation/release-dossier schemas. It does not
contain a populated or signed registry of live contract addresses.

Use it on testnets first. Material mainnet deployment should wait for the release gates in [Security evidence](security-evidence.md), including independent audit and revision-bound deployment verification.

## 2. Immutable decisions

Every constructor choice is permanent. Prepare and independently review a deployment record containing:

- source commit and release tag;
- Solidity, Foundry, Node, and OpenZeppelin versions;
- compiler settings from `foundry.toml`;
- observed RPC chain ID and network name;
- adapter: native or ERC-20;
- exact token address and token due-diligence report if ERC-20;
- protocol name of 1 through 64 bytes;
- distinct nonzero DevCo and security-reserve addresses;
- exactly three verifier addresses in strict numeric ascending order;
- deployment signer and nonce policy;
- intended explorer and source-verification method; and
- finality, monitoring, incident, and deprecation contacts.

The verifier addresses must be unique, must not equal the escrow, DevCo, or the reserve, must have
no code at construction, and must be ordered by their 160-bit address values—not alphabetically by
a label. The code check is only a deployment-time sanity check; later delegation or deployment can
change an address. Acceptance uses ECDSA recovery only. Do not configure an ERC-1271 multisig as a
verifier and assume it will work.

DevCo and reserve are credit recipients, not admins. A contract wallet can receive a credit and call `withdraw` through its own execution path, but this must be tested before deployment.

## 3. Asset selection

For the native adapter, confirm the selected network's native-currency and transaction semantics.

For the ERC-20 adapter, review at least:

- deployed address and chain ID;
- runtime code and proxy implementation/upgrade authority;
- exact transfer and transfer-from balance behavior;
- fee, burn, rebase, hook, callback, pause, freeze, blacklist, blocklist, and confiscation behavior;
- `balanceOf` honesty and revert behavior;
- decimals for display, without using decimals as identity;
- issuer and bridge dependencies; and
- prior production behavior.

The supported profile is an immutable exact-transfer token. Reject fee-on-transfer, rebasing, callback-bearing, deceptive, blacklistable, paused, or upgrade-mutated tokens. Passing one test transfer is not sufficient if an issuer can change behavior later.

Deploy one ERC-20 escrow per token address. Never mix assets in one deployment.

## 4. Clean build and verification

Use a fresh checkout and record all versions:

```sh
node --version
npm --version
forge --version
cast --version
python3 --version
make bootstrap
make check
```

`make check` enforces formatting, a clean size-reporting build, high/medium Forge lint, all unit,
fuzz, and native/ERC-20 invariant tests, the bounded model, semantic model mutations, 23 targeted
Solidity source mutations, deployment/schema syntax, and a two-clean-build bytecode comparison. CI
runs a pinned Slither gate separately. Halmos receipts are not generated by this target and must be
handled explicitly for a release candidate.

Build from an empty generated-artifact directory so removed source files cannot leave stale receipts.

Review the full dependency tree and compiler standard JSON. Do not change optimizer, EVM version, metadata, IR, compiler, or OpenZeppelin version between review and deployment.

## 5. Checked-in Foundry deployment workflow

Store public constructor parameters in a reviewed release record. Keep signing secrets out of shell
history, environment files, and the repository. The checked-in wrapper parses a deliberately small
`KEY=VALUE` data format; it never sources or executes the file. It rejects unknown and duplicate
keys, shell syntax, secret-looking key names, and an `ACCOUNT` entry. An encrypted Foundry keystore
alias is accepted only as the explicit broadcast command-line argument.

Copy the template and replace every value:

```sh
cp .env.example .env
```

Set `NETWORK_KEY` to one exact entry in `deployments/networks.json`, set its catalogued
`EXPECTED_CHAIN_ID`, and set `RPC_URL` to the independently reviewed endpoint. Also set:

- `EXPECTED_SOURCE_REVISION` to the full lowercase 40-hex commit being released;
- `EXPECTED_DEPLOYER` to the dedicated signer address;
- `EXPECTED_DEPLOYER_NONCE` to the reviewed pending nonce;
- `PROTOCOL_NAME`, DevCo, reserve, and the three numerically sorted verifiers; and
- `TOKEN` to zero for native deployment or the reviewed nonzero token for ERC-20.

The wrapper requires `HEAD` to equal the expected revision and refuses a dirty or untracked
worktree. It verifies the reviewed Foundry build revision and compiler settings, performs an
offline build, queries chain ID and the deployer's `pending` nonce with `cast`, and calculates the
expected standard `CREATE` address. The RPC URL and keystore alias are not printed. Do not encode
RPC credentials as URL userinfo, and do not put a private key, mnemonic, password, or shell
expression in `.env`. The deployer, DevCo, reserve, and verifiers must be pairwise distinct; for an
ERC-20 deployment, the token address must also differ from the deployer and fee recipients.

Start with the plan-only target. It performs every preflight and prints the revision, tree,
deployer, pending nonce, and predicted address, but does not run the deployment script:

```sh
make deploy-native-plan ENV_FILE=.env
# or
make deploy-erc20-plan ENV_FILE=.env
```

Then simulate. The expected deployer is bound with `--sender`, so simulation and broadcast derive
the same address when the pending nonce remains unchanged:

```sh
make deploy-native-dry-run ENV_FILE=.env
# or
make deploy-erc20-dry-run ENV_FILE=.env
```

Inspect the simulated constructor configuration, sender behavior, gas, chain ID, compilation output, and logged deployment fields. Have a second operator compare them with the approved record.

Broadcast only by choosing the explicit broadcast Make target and passing a reviewed encrypted
Foundry keystore alias. The target invokes the wrapper with `--execute`; direct wrapper invocation
without `--simulate` or `--execute` remains plan-only. The wrapper rechecks the pending nonce
immediately before handing control to Foundry and also rechecks the source revision/tree and chain
ID. It aborts if any of that context changed:

```sh
make deploy-native ENV_FILE=.env ACCOUNT=reviewed-deployer-keystore-name
# or
make deploy-erc20 ENV_FILE=.env ACCOUNT=reviewed-deployer-keystore-name
```

The Make targets invoke `DeployNative.s.sol` or `DeployERC20.s.sol` with the reviewed sender,
`--broadcast --slow`, and no raw signing secret. A nonce can change after any preflight because of
another process or pending transaction; stop and prepare a new reviewed record rather than merely
incrementing the value.

For a hardware wallet or chain-specific transaction mode, preserve the plan and simulation
receipts, independently re-run the same chain-ID, pending-nonce, and predicted-address checks, then
run the equivalent reviewed `forge script` command with the same explicit `--sender` plus the
appropriate Foundry hardware-wallet flags such as `--ledger` or `--trezor`. Do not turn the `.env`
file into a shell script for this manual path. Hardware-wallet broadcast is intentionally not
automated by the wrapper because device prompts and chain-specific transaction settings require a
human comparison against the approved release record.

The deployment scripts log variant, chain, deployed address, immutable configuration, deployment
ID, verifier-set hash, creation-code hash, constructor-arguments hash, and runtime-code hash.

Verify immediately afterward from the preserved standard JSON through the target explorer or Sourcify. Some networks require legacy transaction settings; decide that from the target network rather than adding it globally.

Do not send native value to either constructor.

## 6. Post-deployment checks

Set the receipt address only after the deployment transaction is finalized:

```sh
export ESCROW='0x...'

cast chain-id --rpc-url "$RPC_URL"
cast code "$ESCROW" --rpc-url "$RPC_URL"
cast call "$ESCROW" 'PROTOCOL_VERSION()(string)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'PROTOCOL_ID()(bytes32)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'deploymentId()(bytes32)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'asset()(address)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'devCo()(address)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'securityReserve()(address)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'verifierSetHash()(bytes32)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'verifierAt(uint256)(address)' 0 --rpc-url "$RPC_URL"
cast call "$ESCROW" 'verifierAt(uint256)(address)' 1 --rpc-url "$RPC_URL"
cast call "$ESCROW" 'verifierAt(uint256)(address)' 2 --rpc-url "$RPC_URL"
cast call "$ESCROW" 'FIXED_FEE_BPS()(uint16)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'MIN_VERIFIER_BPS()(uint16)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'MAX_VERIFIER_BPS()(uint16)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'MAX_BOUNTY_DURATION()(uint64)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'nextBountyId()(uint256)' --rpc-url "$RPC_URL"
cast call "$ESCROW" 'isSolvent()(bool)' --rpc-url "$RPC_URL"
```

For the ERC-20 adapter, also query `token()` and compare it byte-for-byte with the reviewed asset address.

Then set `DEPLOYMENT` in `.env`, keep `TOKEN` zero for native or equal to the deployed ERC-20
asset, and set `EXPECTED_RUNTIME_CODE_HASH` from an independently reproduced exact build—not by
blindly copying the target RPC's answer. Replay the checked-in audit:

```sh
make audit-deployment ENV_FILE=.env
```

This target uses the same strict non-executable parser, clean-revision check, catalogued chain-ID
check, and reviewed toolchain gate as deployment. It does not source `.env`.

`AuditDeployment.s.sol` verifies code exists, requires the observed runtime hash to equal that
independent expectation, rechecks asset, fee recipients, all verifiers, verifier-set hash, and
deployment ID, then logs the code and constructor hashes. It does not verify explorer publication,
transaction provenance, token safety, or the source revision by itself.

Verify source and constructor arguments through at least one public verification service and independently compare actual runtime bytecode with an exact build that substitutes the deployed immutable values. A compiler template runtime hash is insufficient because immutables alter deployed code.

After the worktree is committed and clean, set `NETWORK_KEY`, `DEPLOYMENT_TX_HASH`,
`CONFIRMATIONS`, `MANIFEST_OUTPUT`, and optionally `SOURCE_REPOSITORY`, then run:

```sh
make generate-manifest ENV_FILE=.env
```

This target also passes parsed values to the generator without sourcing `.env` or placing the RPC
URL in a subprocess argument.

The generator refuses a dirty worktree or revision mismatch. It checks the selected chain,
finalized creation receipt, top-level creation transaction, exact locally built creation-bytecode
prefix, decoded constructor arguments, EIP-712 protocol name/domain, reconstructed protocol and
deployment IDs, immutable configuration, independently supplied runtime hash, ERC-20 code hash,
and pinned toolchain/dependency settings. It records block, transaction-input, runtime, creation,
constructor-argument, ABI, dependency-tree, and binary hashes under
`deployments/manifest.schema.json`.

That output is intentionally labeled `unsigned-observation`. It is not a production activation
claim. Build a separately reviewed dossier conforming to
`deployments/release-dossier.schema.json` only after source/explorer exact-match verification,
asset and issuer review, a stated finality policy, full testnet smoke transactions, a reproducible
frontend release, and reviewer signatures. No signing tool or live entry is claimed here.

## 7. Testnet smoke test

Use a small amount and exercise the full lifecycle before enabling a deployment in any frontend:

1. Choose a verifier pool from the profile's expected effort, confirm it lies between
   `minimumVerifierFee(reward)` and `maximumVerifierFee(reward)`, then query
   `requiredFunding(reward, verifierFee)`.
2. Create a bounty and confirm the `BountyCreated` event and stored fields.
3. Submit commitments from two different solver addresses and replace one commitment.
4. Confirm commits fail at the exact commit boundary.
5. Have two configured verifiers reproduce the evaluator and sign the exact EIP-712 record.
6. Relay a winning claim from a third address and verify every credit.
7. Withdraw each kind of credit to tested destinations.
8. Create a second bounty, let it expire, trigger refund from an unrelated relayer, and withdraw the complete funded amount.
9. Repeat for every adapter and token on every target network.
10. Wait for the network's defined finality at each stage and compare indexer state with direct RPC reads.

Do not use production verifier keys or material funds for initial smoke tests.

## 8. Bounty-package operations

Before a sponsor creates a bounty, publish a content-addressed package that freezes:

- objective job statement and acceptance predicate;
- canonical specification and terms bytes;
- profile identifier and evaluator version/hash;
- deterministic input/output and result-digest rules;
- resource and time bounds;
- artifact availability and mirrors;
- disclosure and confidentiality policy;
- commit and claim deadlines in UTC plus raw Unix seconds;
- reward, asset, explicit verifier pool, fixed fees, and full required funding; and
- exact refund recipient.

Hash canonical bytes, not a mutable webpage. Display the resulting `bytes32` values and have the sponsor verify them in its wallet transaction.

The contract does not know a URI or CID. The client must preserve the mapping from each on-chain hash to retrievable bytes for the full bounty life and a reasonable audit period afterward.

## 9. Solver workflow

1. Verify chain ID, escrow address, asset, status, terms hashes, deadlines, reward, and verifier set directly from chain.
2. Produce canonical result bytes and their nonzero digest.
3. Generate a cryptographically random, nonzero, single-use `bytes32` salt and back it up privately.
4. Compute the commitment locally as
   `keccak256(abi.encode(COMMITMENT_TYPEHASH, deploymentId, bountyId, solver, resultDigest, salt))`.
   Never send the result digest or salt to an RPC merely to calculate this hash.
5. Submit the commitment from the intended solver address before the boundary and wait for the required finality.
6. Wait until the commit deadline, then deliver evaluator input to verifiers according to the
   package policy. Revealing it before the boundary can let another solver commit copied work.
7. Obtain two signatures, retain the raw result and receipts, and submit or relay the claim early
   enough to reach the chain's defined finality margin before the half-open claim window closes.
   V1 has no post-deadline grace period: a late claim removed by a reorganization after the claim
   deadline cannot be re-included, and the bounty can then be refunded.
8. After settlement, query `claimable(solver)` and withdraw to a tested destination.

The solver must not be one of the three verifier addresses. Replacing a commitment invalidates the earlier result/salt pair.

## 10. Verifier signing workflow

Before signing, each verifier must independently check:

- RPC chain ID and finalized deployment address;
- `deploymentId`, verifier index/address, and verifier-set hash;
- the exact ascending two-verifier pair and its signed `signerBitmap` (`3`, `5`, or `6`);
- bounty `Open` status and current phase;
- stored solver commitment equals the recomputed commitment;
- frozen profile, specification, terms, and deadline;
- canonical result bytes and `resultDigest`; and
- evaluator result under the documented environment and limits.

Sign the `AcceptedResult` EIP-712 typed data with domain name equal to the constructor protocol name, version `"1"`, selected chain ID, and escrow verifying contract. Do not use `personal_sign` or an Ethereum signed-message prefix. Signatures must be valid canonical ECDSA signatures accepted by OpenZeppelin's recovery rules.

Prepare the EIP-712 payload and digest locally from a finalized block after reading only public
deployment, bounty, and stored-commitment state at that same block. The convenience
`attestationDigest(result, signerBitmap)` call exposes result and salt
to the queried RPC and is therefore inappropriate when that RPC is not trusted with the claim
preimage. It also does not verify that the supplied result matches the stored commitment or that
the bounty is currently claimable. Verifier software must perform those checks before signing.

The two submitted verifier indices must be sorted in ascending numeric index order. The signing service should generate a revision-bound receipt containing evaluator inputs, outputs, hashes, typed data, digest, signature, timestamp, and software identity without logging secret keys.

## 11. Monitoring

Index and alert on:

- `BountyCreated`, `SolverCommitted`, `BountyPaid`, `SettlementRecorded`, `BountyRefunded`, and
  `Withdrawal`;
- approaching commit and claim deadlines;
- open funded amount and claimable credits;
- `isSolvent()`, `accountedBalance()`, `actualBalance()`, and `surplus()`;
- verifier availability and unexpected signing activity;
- token proxy upgrades, pauses, blacklist changes, or balance anomalies;
- chain reorg/finality regressions; and
- frontend registry or source-verification mismatches.

Read critical state directly from multiple RPC operators. An indexer is a cache, not the settlement authority.

## 12. Incident and deprecation procedure

Because there is no pause, upgrade, verifier rotation, rescue, or migration:

1. mark the affected chain/address as deprecated in every release channel;
2. stop the frontend from offering new bounty creation there while retaining read and safe withdrawal/refund access;
3. publish a signed incident notice with exact addresses and observed facts;
4. advise token sponsors to revoke unused approvals;
5. tell users which encoded actions remain safe without promising intervention the contract cannot perform;
6. preserve logs, source, bytecode, receipts, and chain evidence; and
7. deploy a new separately audited instance only after the cause is understood.

Never imply that an operator can return funds early or change a terminal outcome.

## 13. Launch checklist

- [ ] License selected and applied.
- [ ] Final source revision frozen and independently reviewed.
- [ ] `make check` and the pinned CI-equivalent gates pass from a clean checkout.
- [ ] Clean compile, lint, tests, model, mutation, and static analysis pass with preserved receipts.
- [ ] Independent audit completed and findings resolved.
- [ ] Off-chain profile/evaluator and verifier service reviewed.
- [ ] Verifier keys independently secured and recovery/incident plan rehearsed.
- [ ] DevCo and reserve custody tested.
- [ ] Asset behavior and upgrade/issuer risk approved.
- [ ] Checked-in deployment and audit scripts tested from a clean checkout.
- [ ] Full lifecycle smoke-tested on the corresponding testnet.
- [ ] Actual runtime and constructor arguments independently verified.
- [ ] Unsigned deployment observation regenerated from the clean release revision.
- [ ] Signed release dossier published through multiple independent channels.
- [ ] Finality-aware monitoring and incident communication live.
- [ ] Static IPFS frontend reproducibly built and separately reviewed; any indexer treated as an untrusted cache.
- [ ] Initial mainnet limits kept deliberately small after all prior gates.
